## 1. Что мы создаём

В БД `ai_agent` появятся 3 новые таблицы:

| Таблица | Назначение | Кол-во строк (тест) |
|---------|------------|---------------------|
| `appeal_employees` | Реестр сотрудников: ФИО + табельный номер | 60 |
| `appeal_event1` | Журнал **Мероприятия №1**: для каждого сотрудника — выполнено или нет, и когда | 38 |
| `appeal_event2` | Журнал **Мероприятия №2**: аналогично event1 | 23 |

Имена таблиц соответствуют запросам в workflow `Workflow/organization-appeal.json`. Менять их без правки workflow **нельзя**.

> ⚡ **Word-файлы (.docx) обрабатываются прямо в браузере через JSZip** — OCR-сервис для них не требуется. PDF (и старый формат .doc) по-прежнему идут через OCR.

## 2. Логика алгоритма

Алгоритм идентификации проходит 3 ступени. На каждой может произойти ранний выход:

```
┌─────────────────────────────────────────────────────────────────┐
│   1. Документ обращения                                          │
│      .docx → JSZip в браузере → текст                           │
│      .pdf/.doc → OCR-сервис → текст                             │
│      → LLM извлекает ФИО, дату рождения, табельный номер        │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│   ШАГ 1.  appeal_employees                                       │
│   Ищем ФИО в реестре. Сравниваем табельный номер.                │
│                                                                  │
│   ФИО НЕ найдено   → ❌ «Невозможно идентифицировать»             │
│   ФИО найдено, № не совпал → ⚠️ «Идентификация прошла частично»  │
│   ФИО + № совпали  → ✅ идём дальше                              │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│   ШАГ 2.  appeal_event1                                          │
│   Ищем ФИО в журнале Мероприятия №1.                            │
│                                                                  │
│   ФИО НЕ найдено   → ⚠️ «Сведения о данном ФИО отсутствуют»      │
│   is_done = FALSE  → ⚠️ «Мероприятие №1 не выполнено»            │
│   is_done = TRUE   → ✅ идём дальше                              │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│   ШАГ 3.  appeal_event2                                          │
│   Ищем ФИО в журнале Мероприятия №2.                            │
│                                                                  │
│   ФИО НЕ найдено   → ⚠️ «Сведения о Мероприятии №2 отсутствуют»  │
│   is_done = FALSE  → ⚠️ «Мероприятие №2 не выполнено»            │
│   is_done = TRUE   → ✅ «Для данного ФИО проводится проверка»    │
└─────────────────────────────────────────────────────────────────┘
```

## 4. Создание таблиц

Скопируй и выполни блок целиком в `psql`:

```sql
-- =============================================================
-- ТАБЛИЦА 1: Реестр сотрудников
-- =============================================================
CREATE TABLE IF NOT EXISTS appeal_employees (
    id              SERIAL PRIMARY KEY,
    full_name       VARCHAR(200) NOT NULL,
    employee_number VARCHAR(50)  NOT NULL,
    created_at      TIMESTAMP    DEFAULT NOW()
);

-- ILIKE по ФИО — основной запрос алгоритма (шаг 1)
CREATE INDEX IF NOT EXISTS idx_appeal_employees_name
    ON appeal_employees (full_name);

-- Поиск по табельному номеру (для администратора)
CREATE INDEX IF NOT EXISTS idx_appeal_employees_number
    ON appeal_employees (employee_number);

-- Уникальность: один человек = одна запись (опционально, можно убрать)
CREATE UNIQUE INDEX IF NOT EXISTS idx_appeal_employees_unique
    ON appeal_employees (full_name, employee_number);


-- =============================================================
-- ТАБЛИЦА 2: Журнал Мероприятия №1
-- =============================================================
CREATE TABLE IF NOT EXISTS appeal_event1 (
    id          SERIAL PRIMARY KEY,
    full_name   VARCHAR(200) NOT NULL,
    is_done     BOOLEAN      NOT NULL DEFAULT FALSE,
    done_at     TIMESTAMP,
    created_at  TIMESTAMP    DEFAULT NOW()
);
CREATE INDEX IF NOT EXISTS idx_appeal_event1_name
    ON appeal_event1 (full_name);


-- =============================================================
-- ТАБЛИЦА 3: Журнал Мероприятия №2
-- =============================================================
CREATE TABLE IF NOT EXISTS appeal_event2 (
    id          SERIAL PRIMARY KEY,
    full_name   VARCHAR(200) NOT NULL,
    is_done     BOOLEAN      NOT NULL DEFAULT FALSE,
    done_at     TIMESTAMP,
    created_at  TIMESTAMP    DEFAULT NOW()
);
CREATE INDEX IF NOT EXISTS idx_appeal_event2_name
    ON appeal_event2 (full_name);


-- Проверим
\dt appeal_*
```

После выполнения `\dt appeal_*` должны быть видны все 3 таблицы.

## 6. Проверка готовности

После заполнения выполни:

```sql
-- 1. Кол-во строк
SELECT 'appeal_employees' AS table_name, COUNT(*) AS rows FROM appeal_employees
UNION ALL
SELECT 'appeal_event1', COUNT(*) FROM appeal_event1
UNION ALL
SELECT 'appeal_event2', COUNT(*) FROM appeal_event2;
-- Ожидаемо: 60 / 38 / 23

-- 2. Распределение event1 по статусам
SELECT is_done, COUNT(*) FROM appeal_event1 GROUP BY is_done;
-- Ожидаемо: TRUE = 26, FALSE = 12

-- 3. Распределение event2 по статусам
SELECT is_done, COUNT(*) FROM appeal_event2 GROUP BY is_done;
-- Ожидаемо: TRUE = 14, FALSE = 9

-- 4. Сотрудники, прошедшие ВСЕ 3 шага (Мероприятие №1 + №2 выполнены)
SELECT e.full_name, e.employee_number
FROM appeal_employees e
JOIN appeal_event1 ev1 ON ev1.full_name = e.full_name AND ev1.is_done = TRUE
JOIN appeal_event2 ev2 ON ev2.full_name = e.full_name AND ev2.is_done = TRUE
ORDER BY e.full_name;
-- Среди них должен быть «Иванов Иван Иванович» (10001) — главный «зелёный» тест.
```

Если все три счётчика совпали и Иванов виден в JOIN-запросе — всё готово, можно тестировать алгоритм из дашборда.

### Как запустить тест

1. Открой **Дашборд GigaChat → Алгоритмы → Организация обращения**.
2. Перетащи файл из `Tests/OrgAppeal/` в зону загрузки.
3. Нажми «Обработать обращение».
4. Сверь итоговую плашку с колонкой «Ожидаемый итог».

### Если итог отличается между DOCX и PDF

Значит, проблема в одной из ветвей извлечения:

- DOCX прошёл, PDF нет → проблема в **OCR-сервисе** (упал, не читает шрифт, не отвечает)
- PDF прошёл, DOCX нет → проблема в **JSZip-пути** (не подключился `jszip.min.js`, или `extractBrowserText` сломан)

### Перегенерация / новые сценарии

```
# DOCX (без зависимостей — stdlib only)
python Tests/OrgAppeal/generate.py

# PDF (нужен reportlab)
pip install reportlab
python Tests/OrgAppeal/generate-pdf.py
```

`generate-pdf.py` импортирует `TEST_CASES` из `generate.py` — содержимое всегда идентично.

### Добавить сотрудника

INSERT INTO appeal_employees (full_name, employee_number) VALUES

### Отметить выполнение Мероприятия №1

```sql
-- Если записи ещё нет — добавить
INSERT INTO appeal_event1 (full_name, is_done, done_at) VALUES
    ('Новиков Артём Игоревич', TRUE, NOW());

-- Если запись есть, но is_done = FALSE — обновить
UPDATE appeal_event1
SET is_done = TRUE, done_at = NOW()
WHERE full_name = 'Новиков Артём Игоревич';
```

### Снять отметку (на случай ошибки)

```sql
UPDATE appeal_event1
SET is_done = FALSE, done_at = NULL
WHERE full_name = 'Новиков Артём Игоревич';
```

### Найти всех «застрявших» на Мероприятии №1

Сотрудники, у которых есть запись, но мероприятие до сих пор не выполнено:

```sql
SELECT e.full_name, e.employee_number,
       ev1.created_at AS opened_at,
       NOW() - ev1.created_at AS overdue
FROM appeal_employees e
JOIN appeal_event1 ev1 ON ev1.full_name = e.full_name
WHERE ev1.is_done = FALSE
ORDER BY ev1.created_at;
```

### Найти всех «забытых» — нет в event1 вообще

```sql
SELECT e.full_name, e.employee_number
FROM appeal_employees e
LEFT JOIN appeal_event1 ev1 ON ev1.full_name = e.full_name
WHERE ev1.id IS NULL
ORDER BY e.full_name;
```

### Сводка по сотрудникам и обоим мероприятиям

```sql
SELECT
    e.full_name,
    e.employee_number,
    CASE
        WHEN ev1.id IS NULL  THEN 'нет записи'
        WHEN ev1.is_done     THEN 'выполнено'
        ELSE                      'не выполнено'
    END AS event1_status,
    CASE
        WHEN ev2.id IS NULL  THEN 'нет записи'
        WHEN ev2.is_done     THEN 'выполнено'
        ELSE                      'не выполнено'
    END AS event2_status
FROM appeal_employees e
LEFT JOIN appeal_event1 ev1 ON ev1.full_name = e.full_name
LEFT JOIN appeal_event2 ev2 ON ev2.full_name = e.full_name
ORDER BY e.full_name;
```
