# Тестовые документы для алгоритма «Организация обращения»

14 готовых файлов (7 `.docx` + 7 `.pdf` с идентичным содержимым) — по одному на каждую ветвь алгоритма.
Полный список сценариев и ожидаемых результатов — в [`OrgAppeal-Setup.md`](../../OrgAppeal-Setup.md), раздел 7.

## Зачем два формата

- **`.docx`** проверяет браузерный путь — JSZip распаковывает ZIP прямо на клиенте, OCR не задействуется. **Быстро**, не нужен внешний сервис.
- **`.pdf`** проверяет серверный путь — файл уходит на `http://localhost:8055/extract`, OCR-сервис извлекает текст, далее всё как обычно.

Один и тот же сценарий (например, Иванов с табельным 10001) должен дать **одинаковый итог** независимо от формата — это валидация обеих веток workflow.

## Как использовать

1. Открой **Дашборд GigaChat → Алгоритмы → Организация обращения**
2. Перетащи файл в зону загрузки (или выбери кнопкой)
3. Нажми «Обработать обращение»
4. Сверь итог с колонкой «Что проверяем»

Полный прогон — 14 файлов (7 сценариев × 2 формата). На обоих форматах итог должен совпадать.

## Перегенерация

### DOCX (без зависимостей)

```
python generate.py
```

Скрипт использует только stdlib (`zipfile`, `xml`).

### PDF (нужен reportlab)

```
pip install reportlab
python generate-pdf.py
```

`generate-pdf.py` импортирует `TEST_CASES` из `generate.py`, так что содержимое всегда **идентично** docx. Шрифт — Arial из `C:\Windows\Fonts\arial.ttf` (если у тебя не Windows — поменяй `FONT_PATH` в скрипте).

## Структура файлов

### .docx
Минимально валидный OOXML (3 файла внутри ZIP):
- `[Content_Types].xml`
- `_rels/.rels`
- `word/document.xml` (текст в UTF-8 в `<w:t>`)

### .pdf
Стандартный PDF 1.4 с встроенным subset шрифта Arial для кириллицы. Текст реально вписан в PDF (не картинка), поэтому OCR-сервис извлекает его прямо, без распознавания изображения. Размер ~80 КБ из-за embedded font.
