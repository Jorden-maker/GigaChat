# Импорт workflow в n8n — гайд по `import-workflows.ps1`

Скрипт `import-workflows.ps1` в корне проекта массово синхронизирует все `.json` из папки `Workflow/` с n8n через Public API.

Идемпотентный: запускай сколько угодно раз — дубликатов не будет.

## 1. Что делает скрипт

На каждый `.json` в папке `Workflow/`:

```
JSON содержит "name": "..."
        ↓
В n8n уже есть workflow с таким именем?
        ↓                              ↓
       ДА                              НЕТ
        ↓                              ↓
   PUT /workflows/{id}            POST /workflows
   (обновляет тело)               (создаёт новый)
   credentials остаются           
```

Перед отправкой скрипт **фильтрует** JSON — оставляет только поля, которые n8n API принимает (`name`, `nodes`, `connections`, `settings`). Все остальные поля из экспорта (`id`, `meta`, `versionId`, `tags`, `pinData`, `active`) удаляются — n8n их отвергает с 400 Bad Request.

## 2. Когда запускать

| Сценарий                                            | Что произойдёт                |
|-----------------------------------------------------|-------------------------------|
| Первый импорт на чистую n8n                         | `создано: N`, `обновлено: 0` |
| Поправил один JSON в `Workflow/`                    | `создано: 0`, `обновлено: N` |
| Добавил новый JSON в папку                          | `создано: 1`, `обновлено: N` |
| Запустил без изменений                              | `обновлено: N` (безопасно)   |

## 3. Перед первым запуском

Что нужно:
- **PowerShell 5.1+** (стандартный на Windows 10/11) или PowerShell 7.
- **HTTP-доступ к n8n** (на любом ПК в локальной сети, не обязательно на том же).
- **API-ключ от n8n** (создаётся в Settings → API).

## Шаг 1 — получить API-ключ из n8n

1. Открой n8n в браузере.
2. **Settings** (шестерёнка слева внизу) → **API**.
3. Кнопка **Create an API key**.
4. Введи имя (например `bulk-import`).
5. **Скопируй сразу** — показывается один раз, потом не увидишь.

Если потерял — создай новый. Никаких проблем.

> **Безопасность:** ключ даёт полный доступ к API n8n. Не публикуй в чатах и не коммить в git.

## Шаг 2 — поправить переменные в скрипте

Открой `import-workflows.ps1` в Блокноте или редакторе. В самом верху:

```powershell
$folder = "C:\Users\Lenovo\Desktop\GigaChat\Workflow"
$n8n    = "http://localhost:5678"
$apiKey = ""
$prefix = "[GigaChat] "
```

| Переменная | Что подставить                                                                |
|------------|--------------------------------------------------------------------------------|
| `$folder`  | Полный путь к папке `Workflow/` на твоём ПК. Если проект лежит в другом месте — поправь. |
| `$n8n`     | URL n8n. **Без `/` на конце.** Примеры: `http://localhost:5678`, `http://192.168.1.50:5678`. |
| `$apiKey`  | Ключ из Шага 1, **без угловых скобок** `<` и `>`.                              |
| `$prefix`  | Префикс к имени каждого workflow в n8n. По умолчанию `[GigaChat] ` (с пробелом в конце). Чтобы отключить — поставь пустую строку `""`. |

Чуть ниже — блок `$credentialMapping`. **По умолчанию ничего трогать не надо**, скрипт сам резолвит credentials (см. раздел [Привязка credentials](#привязка-credentials)). Менять туда нужно только если:
- У тебя другой URL/apiKey для GigaChat — поправь `openAiApi.data.url` и `apiKey`.
- Хочешь дать credentials другие имена в n8n — поправь `name` у нужного типа.
- Хочешь сам вписать ID существующего credential (а не позволять скрипту создавать) — впиши его в `id`.

Сохрани файл.

### Зачем префикс

n8n в self-hosted версиях не поддерживает папки (Folders) — это платная фича Cloud. Префикс — простой обходной путь: все workflow в общем списке группируются вместе по алфавиту. После импорта в UI они выглядят так:

```
[GigaChat] Чат-Агент. Поток
[GigaChat] Загрузка истории. Поток
[GigaChat] Загрузчик документов. Поток
[GigaChat] Математический агент. Поток
[GigaChat] OCR + Поиск ФИО. Поток
[GigaChat] Поток (Роутер)
[GigaChat] Промпт-Инженер. Поток
[GigaChat] RAG-Агент. Поток
[GigaChat] SQL-Агент. Поток
```

В JSON-файлах на диске имена остаются **без префикса** — префикс добавляется только при отправке в n8n. Это сделано чтобы локальные файлы не загромождать визуально и не зависеть от того, где они потом окажутся.

## Привязка credentials

В `.json`-файлах workflow прописаны ID credentials с **разработческой машины**. На офисной/новой машине таких ID нет, и без подмены каждый импортированный workflow показывал бы «битый credential» в узлах LLM и Postgres. Скрипт автоматически чинит это.

### Что делает скрипт

Перед самым первым импортом скрипт делает **резолвинг credentials** один раз:

```
==> Резолвлю credentials...
  Создаю credential 'GigaChat' (тип openAiApi) в n8n...
    OK: создан, id = a8X2tYpFq...
    openAiApi -> id: a8X2tYpFq..., name: GigaChat
    postgres  -> не резолвлен (узлы останутся с битым credential)
```

Что произошло:
1. **GigaChat (`openAiApi`)** — скрипт сам создал credential в n8n через `POST /api/v1/credentials` с настройками из блока `$credentialMapping.openAiApi.data` (`url`, `apiKey`). Полученный ID запомнил в `credentials-cache.local.json` рядом со скриптом.
2. **Postgres** — `autoCreate = $false` (пароль БД не должен жить в скрипте в git). Пока не привяжешь — узлы Postgres будут битыми.

Дальше при каждом обработанном workflow скрипт **подменяет ID и `name`** в полях `credentials.openAiApi` / `credentials.postgres` каждого узла на резолвленные.

### Что нужно сделать ОДИН раз на новой машине

**Postgres credential** создаётся вручную через UI n8n (потому что нужен пароль БД):
1. n8n → **Credentials** → **Add credential** → **Postgres**.
2. Имя: `Postgres` (важно — то же что в скрипте, иначе подмена не сработает).
3. Заполни host / port / database / user / password.
4. Сохрани.
5. Скопируй ID из URL страницы редактирования (`/credentials/<ID>/edit`).
6. В скрипте впиши его в `$credentialMapping.postgres.id`.

Повторный запуск `import-workflows.ps1` после этого подцепит Postgres ко всем workflow автоматически.

**GigaChat credential** обычно создаётся **скриптом сам** (autoCreate = `$true`). Но если у тебя credential с именем `GigaChat` **уже существует** в n8n, скрипт получит 4xx — тогда:
1. Открой существующий credential в UI n8n.
2. Скопируй ID из URL.
3. Впиши в `$credentialMapping.openAiApi.id`.
4. Запусти скрипт ещё раз.

### Локальный кеш ID

`credentials-cache.local.json` рядом со скриптом — хранит резолвленные ID. **В git не комитится** (см. `.gitignore`).

Когда полезен:
- При повторных запусках скрипт не создаёт credentials заново — берёт ID из кеша.
- Можно переносить с одной машины на ту же сессию n8n (если та же база).

Когда удалить:
- Поменял n8n БД → ID устарели → удали файл, скрипт пересоздаст.
- Случайно создались дубли credentials → разберись в UI, удали лишние, удали кеш, запусти скрипт.

### Активация workflow

После **CREATED** workflow приходит в n8n **выключенным**. Два способа активировать:

**Способ 1 — через скрипт (массово):**
```powershell
.\activate-workflows.ps1
```
Активирует все workflow с префиксом `[GigaChat] ` одной командой. Идемпотентный — уже активные пропускает.

**После правки workflow** (`import-workflows.ps1` обновил уже активный):
```powershell
.\activate-workflows.ps1 -Force
```
Делает `deactivate → activate` для активных workflow — это **перерегистрирует webhook routes** в runtime n8n. Нужно когда изменил URL HTTP-ноды, путь webhook'а или структуру триггеров.

Для деактивации: `.\activate-workflows.ps1 -Deactivate`.

**Способ 2 — вручную (по одному в UI n8n):**
1. Открой workflow.
2. Тумблер **Active** справа сверху → **Save**.

После **UPDATED** активность сохраняется — менять не нужно.

> Цвет всё ещё красный у узла? Значит credential либо не зарезолвился (см. вывод скрипта), либо `name` в `$credentialMapping` не совпадает с тем что в n8n. Проверь оба условия и перезапусти.

## Важно про PUT и UI-правки

PUT **полностью заменяет тело workflow** содержимым из JSON.

Что это значит на практике:

- Если ты двигал узлы мышью в UI n8n — после следующего запуска скрипта они **вернутся** в позицию из JSON.
- Если ты менял параметры узла через UI (например прописал URL, поменял query) — эти изменения **исчезнут**.

**Правило:** держи **источник истины** в одном месте.
- Если работаешь над workflow в **UI** — не запускай скрипт, иначе всё затрётся.
- Если работаешь над workflow в **JSON** — после каждой правки запускай скрипт, синхронизируй в n8n.

Лично рекомендую второй подход: JSON в git → отслеживание истории, легко откатывать, легко переносить между средами (dev / office / production).

Credentials, активность (`active: true/false`) и история выполнений (`executions`) хранятся **отдельно от тела workflow** и PUT не трогает их.

## Решение частых проблем

### `ОШИБКА: переменная $apiKey пустая`

Не подставил ключ в строку `$apiKey = ""`. Открой скрипт, вставь ключ между кавычками.

### `Could not find a part of the path` или `папка не найдена`

`$folder` указывает на несуществующий путь. Проверь, что папка `Workflow/` лежит именно там, где ты написал.

### `FAILED: ... 404 ...`

В `$n8n` остался слеш на конце:
```powershell
$n8n = "http://localhost:5678/"   ← плохо
$n8n = "http://localhost:5678"    ← хорошо
```

### `FAILED: ... 401 ...`

API-ключ неправильный, или в нём остались угловые скобки:
```powershell
$apiKey = "<eyJhbGc...>"   ← плохо
$apiKey = "eyJhbGc..."     ← хорошо
```

Или ключ был удалён в n8n. Создай новый.

### `FAILED: ... 400 ...` со `Server: ...field X is not allowed...`

JSON содержит поле, которое n8n не принимает. Скрипт фильтрует основные (`id`, `meta`, и т.п.), но если в JSON появилось что-то новое — добавь его в массив исключения. Покажи мне сообщение об ошибке, помогу.

### Кириллические имена workflow рендерятся как `???` или `Ð¡` в n8n

В скрипте есть UTF-8-преобразование (`[System.Text.Encoding]::UTF8.GetBytes`), которое решает эту проблему. Если всё равно сломано — убедись что:
- Файл `.json` сохранён в UTF-8.
- Файл скрипта `.ps1` — с UTF-8 BOM (в репо так и есть).

### Скрипт ругается `running scripts is disabled on this system`

Не выполнил Шаг 3. Запусти один раз:
```powershell
Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned
```

### Сломалась кодировка вывода в самом окне PowerShell

Кириллица в `Write-Host` показывается как кракозябры. Это проблема консоли, не скрипта. Перед запуском в этом же окне выполни:
```powershell
chcp 65001
$OutputEncoding = [System.Text.UTF8Encoding]::new()
```

И потом запускай скрипт.

### Где смотреть полный ответ от n8n при ошибке

Скрипт сам печатает тело ошибки от сервера, если есть, жёлтым цветом строкой `Server: ...`. Это и есть полный диагностический текст. Если он не появился — ошибка случилась до отправки запроса (например проблема с PowerShell или путём к файлу).
